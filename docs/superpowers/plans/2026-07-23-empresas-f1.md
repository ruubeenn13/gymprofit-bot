# Empresas · Fase 1 — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Poder fundar una empresa (entidad tipo gremio ligada a rama) y pertenecer a ella por invitación o por solicitud-con-motivo; más `/trabajo dimitir`.

**Architecture:** Migración Flyway V27 (3 tablas) → dominio (enums + records) → `EmpresaRepositorio` (JDBC, estilo `CarreraRepositorio`) → `EmpresaService` (reglas + enums de resultado, estilo `TrabajoService`) → `/empresa` (comando con subcomandos + listener de botones) y `/trabajo dimitir`. Sin economía/rangos/despidos (fases siguientes).

**Tech Stack:** Java 21, JDA 5, Flyway, MySQL (Aiven), JUnit 5 + Mockito, Testcontainers (repos), i18n `messages_es/en.properties`, `EmbedFactory`.

**Spec:** `docs/superpowers/specs/2026-07-23-empresas-f1-design.md`. **Build de referencia:** 446 tests.

**Patrones a imitar (leerlos antes de escribir):**
- Repo JDBC + test Testcontainers: `CarreraRepositorio` + su test.
- Service con enums de resultado y cobro-lo-último: `TrabajoService` (métodos `elegir`/`ascender`).
- Comando con subcomandos + confirmación/botones: `TrabajoComando` y un listener de botones existente (p. ej. el de tickets o el paginador) para el `customId` con estado.
- Cobro quemado: `EconomiaRepositorio.gastar(...)` (misma llamada que usa `ascender`).

CREATE TABLE empresa_miembros (
    empresa_id  BIGINT       NOT NULL,

CREATE TABLE empresa_pendientes (

    empresa_id  BIGINT       NOT NULL,

- [ ] **Step 1: `docs/decisions.md`** — comprueba el último ADR (`grep -n "^## ADR" docs/decisions.md | tail -1`; debería ser 015 → nuevo **ADR-016**):

```markdown
## ADR-016 — empresas como entidad ligada a rama

**Estado:** aceptada e implementada (Fase 1).

**Contexto.** Se pidió una jerarquía de trabajo «como una empresa»: dimitir y que los cargos altos
gestionen a los inferiores. El sistema de ascensos daba carrera individual por rama, pero no una
entidad colectiva.

**Decisión.** Empresa como **entidad** ligada a una rama, no una capa sobre la carrera: la funda un
**t4 de la rama** (coins quemados), un jugador pertenece a **una sola** empresa, y el ingreso exige
**consentimiento** por ambas vías (invitación del dueño / solicitud-con-motivo del jugador). La F1
monta solo estructura y pertenencia; rangos gestionables, ascenso por el jefe, despidos, economía
(reparto/bote/nivel), ranking y las funciones «vida real» (producción, impuestos, acciones,
reputación, bolsa de empleo, préstamos, eventos) quedan para fases posteriores.

**Consecuencias.** Migración V27 (empresas, empresa_miembros, empresa_pendientes). `/empresa` nuevo y
`/trabajo dimitir`. La pertenencia se valida al entrar, no de forma continua (dimitir no expulsa de la
empresa en F1).
```
