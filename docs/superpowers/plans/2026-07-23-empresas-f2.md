# Empresas · Fase 2 (Jerarquía y gestión) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Gestionar la plantilla de una empresa: cambiar rango, sacar (conserva trabajo) y despedir (al paro); con gobernanza — el dueño actúa directo y un directivo propone, resuelto por voto mayoritario de los altos cargos.

**Architecture:** Migración V28 (2 tablas) → dominio → `EmpresaPropuestaRepositorio` + métodos nuevos en `EmpresaRepositorio` → `EmpresaGestionService` (autoridad, ejecución, predicado de votación) → subcomandos en `/empresa` + botones de voto. Sin ascenso-de-tier ni economía (F3+).

**Tech Stack:** Java 21, JDA 5, Flyway/MySQL, JUnit 5 + Mockito, Testcontainers, i18n, `EmbedFactory`.

**Spec:** `docs/superpowers/specs/2026-07-23-empresas-f2-design.md`. **Build de referencia:** 469 tests.

**Patrones a imitar:** `EmpresaRepositorio`/`EmpresaRepositorioTest` (JDBC + Testcontainers), `EmpresaService`/`EmpresaServiceTest` (enums de resultado, Mockito), `EmpresaComando`/`EmpresaBotonesListener` (subcomandos + botones con customId). `RangoEmpresa` (enum en orden ascendente: `ordinal()` = autoridad).

-- altos cargos votan, y los votos. El dueño actúa directo (sin propuesta); estas tablas son para las

    id                     BIGINT       NOT NULL AUTO_INCREMENT,
    empresa_id             BIGINT       NOT NULL,

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci

- [ ] **Step 3: Implementa `EmpresaGestionService`** siguiendo la spec §Servicio y §Gobernanza. Puntos críticos (código exacto del predicado):
```java
/** Aprobada si los Sí son mayoría estricta de los N altos cargos, o hay empate y el Dueño votó Sí. */
private boolean aprobada(int n, int si, boolean duenoVotoSi) {
    return si * 2 > n || (si * 2 == n && duenoVotoSi);
}
private boolean rechazada(int n, int no, boolean duenoVotoNo) {
    return no * 2 > n || (no * 2 == n && duenoVotoNo);
}
```
`gestionar(actorId, tipo, objetivoId, rangoNuevo)`:
  - Carga la empresa del actor (`repo.deMiembro(actorId)`); el objetivo debe ser miembro de la MISMA empresa (`NO_ES_MIEMBRO` si no).
  - Regla de rango: `rangoActor.ordinal() > rangoObjetivo.ordinal()` y, en CAMBIAR_RANGO, `rangoNuevo.ordinal() < rangoActor.ordinal()` y `rangoNuevo != DUENO` (`RANGO_INVALIDO` si falla).
  - Si `rangoActor == DUENO` → ejecuta directo → `EJECUTADA`. Si `rangoActor == DIRECTIVO` → `propuestas.crear(...expira = now()+DURACION...)` + `votar(propId, actorId, true)` → `PROPUESTA_CREADA` (traduce la UNIQUE a `YA_HAY_PROPUESTA`). Otro rango → `NO_AUTORIZADO`.
`votar(propuestaId, votanteId, si)`:
  - Carga propuesta (`NO_EXISTE`); si `now() > expira` → `cerrar` + `CADUCADA`. Votante debe ser alto cargo de esa empresa (`NO_AUTORIZADO`). `propuestas.votar(...)`. Recarga votos + `altosCargos`; calcula `N`, `si`, `no`, si el dueño votó y su sentido. Si `aprobada(...)` → **revalida la regla de rango** y ejecuta la acción, `cerrar`, `APROBADA_EJECUTADA`; si `rechazada(...)` → `cerrar`, `RECHAZADA`; si no → `REGISTRADO`.
  - Ejecución compartida (privada) según `tipo`, como en Step 2.

- [ ] **Step 1: `docs/decisions.md`** — nuevo **ADR-017** (comprueba que el último es 016):
```markdown
## ADR-017 — gobernanza de empresas

**Estado:** aceptada e implementada (Fase 2).

**Contexto.** La F1 daba miembros y rango, pero el rango no se podía cambiar ni echar a nadie. Se
quería «que los cargos altos gestionen a los inferiores», pero sin que un Directivo mande solo.

**Decisión.** El **Dueño** gestiona directo (cambiar rango, sacar, despedir) a cualquier rango
inferior. Un **Directivo** no ejecuta: **propone**, y los **altos cargos** (Dueño + Directivos)
votan; se aprueba por mayoría estricta, con el voto del Dueño como desempate, y caduca a las 48 h.
Echar tiene dos modos: **sacar** (conserva el trabajo) y **despedir** (al paro). La regla de rango
(nunca tocar a un igual/superior) acota el abuso. El disparo de ascenso de tier de un miembro se
**difiere a F3** por su acople con la economía (lo patrocina el bote).

**Consecuencias.** Migración V28 (empresa_propuestas, empresa_votos). `EmpresaGestionService` separado
de `EmpresaService`. La regla de rango se revalida al ejecutar (el objetivo pudo cambiar entre
proponer y aprobar).
```
